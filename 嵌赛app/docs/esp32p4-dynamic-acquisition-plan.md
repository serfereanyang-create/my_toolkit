# ESP32-P4 动态采集接入规划

## 1. 目标定位

后续系统中，ESP32-P4 作为主控节点，负责接入气体传感器数据和视觉识别数据，并把结果上传到管理员监控端。当前静态网站只做演示，后续需要把模拟数据源替换成真实采集数据源。

系统需要同时支持两种运行方式：

1. 电脑调试模式：通过 USB 串口连接 ESP32-P4，电脑端自动识别 COM 口并接入数据。
2. 独立运行模式：ESP32-P4 通过 Wi-Fi 连接网络，参考 ESP-IDF Wi-Fi 和 HTTP/MQTT 示例，将数据主动上传到服务器或云端。

## 2. 总体数据链路

```text
气体传感器 / K230D AI-CAM
        ↓
ESP32-P4 主控采集与整理
        ↓
风险初步判断 / 数据打包
        ↓
电脑调试串口模式 或 Wi-Fi 独立上传模式
        ↓
后端接口 / 云数据库 / 静态前端适配层
        ↓
管理员监控页面实时渲染
```

## 3. 电脑调试模式：串口自动识别 COM 口

### 3.1 使用场景

该模式适用于开发、调试和比赛现场演示时连接电脑使用。ESP32-P4 通过 USB 连接电脑，电脑端程序自动扫描可用串口，识别 ESP32-P4 对应的 COM 口，并读取串口输出的数据。

### 3.2 数据输出方式

ESP32-P4 固件通过串口周期性输出一行 JSON 数据。每行代表一次采集快照。

示例：

```json
{
  "deviceId": "lab-node-001",
  "timestamp": 1770000000000,
  "source": "uart",
  "sensors": {
    "co": 12,
    "alcohol": 180,
    "voc": 220
  },
  "vision": {
    "status": "normal",
    "label": "未发现异常",
    "confidence": 0.96
  },
  "actuators": {
    "fan": "off",
    "alarm": "off"
  }
}
```

注意：字段可以后续继续改，目前只要求保持“一行一个 JSON 快照”的输出习惯，方便电脑端解析。

### 3.3 电脑端自动识别 COM 口策略

电脑端可以使用 Node.js、Python 或 Electron 实现串口桥接程序。

建议识别逻辑：

1. 扫描系统所有串口。
2. 根据 USB VID/PID、设备描述、制造商信息初步筛选 ESP32 相关端口。
3. 对候选 COM 口依次打开。
4. 等待 1~3 秒串口输出。
5. 如果读到合法 JSON，且包含 deviceId 或约定字段，则认为接入成功。
6. 接入成功后持续读取数据并转发给网页或本地后端。

### 3.4 电脑调试模式推荐架构

```text
ESP32-P4 串口 JSON 输出
        ↓
电脑端串口桥接程序
        ↓
本地 WebSocket / HTTP 服务
        ↓
网页实时显示
```

推荐使用方式：

- 串口桥接程序读取 ESP32-P4 数据。
- 桥接程序在电脑本地开一个 WebSocket 服务。
- 静态网页连接 `ws://localhost:xxxx` 获取实时数据。
- 如果未来部署公网，可以让桥接程序把数据同时 POST 到公网后端。

### 3.5 电脑调试模式优点

- 不依赖 Wi-Fi，调试稳定。
- 方便查看原始串口日志。
- 适合传感器标定、阈值调试、视觉结果验证。
- 比赛现场如果网络环境不好，可以使用串口模式兜底。

## 4. 独立运行模式：Wi-Fi 上传

### 4.1 使用场景

该模式用于项目最终独立运行。ESP32-P4 不依赖电脑，通过 Wi-Fi 连接路由器或手机热点，然后将采集到的数据上传到后端服务或云平台。

### 4.2 ESP-IDF 参考方向

后续固件开发可参考 ESP-IDF 中以下类型示例：

1. Wi-Fi Station 示例：用于让 ESP32-P4 连接路由器。
2. Wi-Fi SmartConfig / Provisioning 示例：用于后续配网。
3. HTTP Client 示例：用于向后端接口发送 HTTP POST。
4. MQTT Client 示例：如果后续需要更实时的发布订阅模式。
5. NVS 示例：用于保存 Wi-Fi 账号、密码、设备编号和阈值配置。

### 4.3 Wi-Fi 上传方式选择

#### 方案 A：HTTP POST

推荐前期优先使用。

```text
ESP32-P4 → HTTP POST → 后端 API → 数据库存储 → 前端查询/刷新
```

优点：

- 实现简单。
- 方便调试。
- 和当前项目接口骨架兼容。
- 适合比赛展示。

缺点：

- 实时性略弱于 MQTT/WebSocket。

#### 方案 B：MQTT

适合后期优化。

```text
ESP32-P4 → MQTT Broker → 后端订阅 → 前端实时显示
```

优点：

- 实时性更好。
- 适合多设备扩展。

缺点：

- 架构复杂度更高。
- 需要维护 MQTT Broker 或云服务。

### 4.4 独立模式建议流程

1. ESP32-P4 启动。
2. 从 NVS 读取 Wi-Fi 配置。
3. 连接 Wi-Fi。
4. 初始化气体传感器和视觉数据输入。
5. 周期性采集 CO、酒精气体、VOC、视觉识别结果。
6. 本地初步判断风险等级。
7. 控制风机、蜂鸣器、报警灯等执行器。
8. 将数据通过 HTTP POST 上传。
9. 如果上传失败，则本地缓存最近若干条数据，等待网络恢复后补传。

## 5. 数据适配层设计

由于后续数据定义一定会变化，网页端不要直接绑定 ESP32-P4 原始字段。建议增加一层数据适配层。

```text
ESP32-P4 原始数据
        ↓
adapter 适配层
        ↓
页面统一快照模型
        ↓
渲染模块
```

页面统一快照模型只保留页面关心的信息块：

- 系统风险等级
- 气体传感器列表
- 视觉识别状态
- 执行器状态
- 设备状态
- 报警事件
- 趋势数据
- 事件流

这样后续即使 ESP32-P4 字段变化，也只需要改 adapter，不需要重写页面。

## 6. 后续开发顺序建议

1. 先保留当前静态网页模拟数据。
2. 增加一个“数据源模式”开关：模拟数据 / 串口桥接 / Wi-Fi 云端。
3. 开发电脑端串口桥接程序，实现自动识别 COM 口。
4. ESP32-P4 固件先实现串口 JSON 输出。
5. 网页接入本地 WebSocket 数据源。
6. ESP32-P4 固件再实现 Wi-Fi Station 和 HTTP POST 上传。
7. 后端接收 ESP32-P4 上传数据，并存储到 Supabase 或其他数据库。
8. 前端从后端读取数据，实现公网监控。

## 7. 说明书中可写的表述

本系统的软件通信设计支持调试与独立运行两种模式。在调试模式下，ESP32-P4 通过 USB 串口向电脑端输出气体传感器和视觉识别数据，电脑端程序自动扫描并识别对应 COM 口，实现快速接入与本地调试。在独立运行模式下，ESP32-P4 参考 ESP-IDF Wi-Fi Station、HTTP Client 等示例，通过 Wi-Fi 接入网络并将采集数据上传至服务器，管理员可通过网页端查看实时状态、报警事件和历史记录。该设计兼顾开发调试便利性和系统独立部署能力，为后续多节点扩展和公网监控提供基础。
